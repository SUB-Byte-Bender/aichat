# Mobile Context Menu Implementation

## Problem Description

**Issue**: Context menus triggered by right-click (desktop) don't work on mobile/touch devices. Users need a way to access "Rename" and "Delete" options for chat items on mobile.

**Challenge**: Mobile browsers don't have a native "right-click" equivalent, so we need to implement a touch-based alternative.

## Solution: Long-Press Gesture

Implement a long-press gesture (500ms hold) that triggers the context menu at the touch location.

## Implementation Steps

### Step 1: Add State and Ref for Timer

In the component where context menu is used (e.g., `AppSidebar`):

```tsx
const longPressTimer = useRef<any>(null);
```

**Why `any` type?** Avoids TypeScript issues between browser `setTimeout` (returns `number`) and Node.js `setTimeout` (returns `NodeJS.Timeout`).

### Step 2: Attach Touch Handlers to Parent Element

**CRITICAL**: Don't attach touch handlers to wrapped components (e.g., components wrapped in `Tooltip`, `Popover`, etc.). Attach them to a simple parent element that forwards all props.

✅ **Correct** - Attach to `SidebarMenuItem` (simple `<li>` element):
```tsx
<SidebarMenuItem 
  onTouchStart={(e) => {
    const touch = e.touches[0];
    longPressTimer.current = setTimeout(() => {
      setContextMenu({ x: touch.clientX, y: touch.clientY, chatId: chat.id });
    }, 500);
  }}
  onTouchEnd={() => {
    if (longPressTimer.current) {
      clearTimeout(longPressTimer.current);
      longPressTimer.current = null;
    }
  }}
  onTouchMove={() => {
    if (longPressTimer.current) {
      clearTimeout(longPressTimer.current);
      longPressTimer.current = null;
    }
  }}
>
  <SidebarMenuButton {...props} />
</SidebarMenuItem>
```

❌ **Incorrect** - Attaching to wrapped component:
```tsx
<SidebarMenuButton 
  onTouchStart={...}  // Won't work if wrapped in Tooltip
  {...props}
/>
```

### Step 3: Fix Z-Index Issues

**Problem**: Context menu might be hidden behind other elements (like mobile sidebar sheets).

**Solution**: Ensure the context menu has a higher z-index than overlays.

```tsx
<motion.div
  className="fixed z-[100] glass overflow-hidden min-w-[150px]"
  style={{ left: contextMenu.x, top: contextMenu.y }}
>
  {/* Menu items */}
</motion.div>
```

**Common z-index hierarchy**:
- Mobile sidebar/sheets: `z-50`
- Context menu: `z-[100]`
- Modals: `z-50` to `z-[100]`

### Step 4: Fix Pointer Events

**Problem**: Even with correct z-index, clicks might pass through the menu to elements behind it.

**Solution**: Add `pointer-events-auto` to the context menu:

```tsx
<motion.div
  className="fixed z-[100] glass overflow-hidden min-w-[150px] pointer-events-auto"
>
```

**Why this is needed**: Some CSS frameworks or parent elements might have `pointer-events: none`, causing events to pass through.

## Complete Example

```tsx
// State & Ref
const [contextMenu, setContextMenu] = useState<{ x: number; y: number; chatId: string } | null>(null);
const longPressTimer = useRef<any>(null);

// Handler functions
const handleContextMenu = (e: React.MouseEvent, chatId: string) => {
  e.preventDefault();
  e.stopPropagation();
  setContextMenu({ x: e.clientX, y: e.clientY, chatId });
};

// In JSX
<SidebarMenuItem 
  onTouchStart={(e) => {
    const touch = e.touches[0];
    longPressTimer.current = setTimeout(() => {
      setContextMenu({ x: touch.clientX, y: touch.clientY, chatId: chat.id });
    }, 500);
  }}
  onTouchEnd={() => {
    if (longPressTimer.current) {
      clearTimeout(longPressTimer.current);
      longPressTimer.current = null;
    }
  }}
  onTouchMove={() => {
    if (longPressTimer.current) {
      clearTimeout(longPressTimer.current);
      longPressTimer.current = null;
    }
  }}
>
  <SidebarMenuButton
    onClick={() => setCurrentChatId(chat.id)}
    onContextMenu={(e) => handleContextMenu(e, chat.id)}
  >
    {/* Button content */}
  </SidebarMenuButton>
</SidebarMenuItem>

{/* Context Menu Rendering */}
<AnimatePresence>
  {contextMenu && (
    <motion.div
      className="fixed z-[100] glass overflow-hidden min-w-[150px] pointer-events-auto"
      style={{ left: contextMenu.x, top: contextMenu.y }}
    >
      <button onClick={handleEdit}>Edit</button>
      <button onClick={handleDelete}>Delete</button>
    </motion.div>
  )}
</AnimatePresence>
```

## Troubleshooting Guide

### Issue 1: Touch handlers not firing
**Cause**: Handlers attached to a component that doesn't forward touch event props (e.g., wrapped in Tooltip).  
**Fix**: Move handlers to a simpler parent element like `<div>` or `<li>`.

### Issue 2: Context menu not visible
**Cause**: Z-index too low, hidden behind other overlays.  
**Fix**: Increase z-index to `z-[100]` or higher.

### Issue 3: Menu visible but not clickable
**Cause**: Missing `pointer-events-auto` or parent has `pointer-events: none`.  
**Fix**: Add `pointer-events-auto` to the menu element.

### Issue 4: Menu triggers while scrolling
**Cause**: Missing `onTouchMove` handler to cancel the timer.  
**Fix**: Add `onTouchMove` handler that clears the timer.

## Key Learnings

1. **Event Forwarding**: Not all React components forward all event props. When in doubt, use simpler parent elements.

2. **Z-Index Hierarchy**: Always maintain a clear z-index hierarchy. Document the levels used in your app.

3. **Pointer Events**: When dealing with overlays, always consider pointer events. Default is usually `auto`, but some libraries/frameworks may override this.

4. **Timer Cleanup**: Always clean up timers in both `onTouchEnd` and `onTouchMove` to prevent memory leaks and unwanted behavior.

5. **Touch vs Click**: Touch events (`onTouchStart`, `onTouchEnd`, `onTouchMove`) are separate from mouse events. Both need to be handled for full cross-device support.
