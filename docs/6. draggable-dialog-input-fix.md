# Draggable Dialog Input Selection Fix

## Problem Description

**Issue**: When using Framer Motion's `drag` prop on a dialog container, interacting with child `Input` or `Textarea` elements triggers the drag gesture instead of allowing text selection or focus.

**Symptoms**:
- Clicking the input drags the dialog.
- Cannot select text inside the input.
- "Not smooth" dragging when trying to use manual `dragControls`.

## The Challenge

1.  **Smoothness vs. Control**: Using `dragListener={false}` with manual `dragControls` (triggered by the header) solves the selection issue but often results in "laggy" or "not smooth" dragging behavior compared to the native `drag` prop.
2.  **Event Propagation**: Framer Motion's drag listeners are often attached as native listeners. React's synthetic `onPointerDown={(e) => e.stopPropagation()}` is sometimes insufficient because the native event might reach Framer Motion's listener before React processes the synthetic event, or due to how Framer Motion handles event capture.

## The Solution

The robust solution involves two parts:
1.  **Enable Native Drag**: Use the standard `drag` prop on the `motion.div` to ensure smooth, native-feeling dragging.
2.  **Native Event Blocking**: Use a `ref` on the input's wrapper to attach **native DOM event listeners** that stop propagation. This guarantees the event is killed before it bubbles up to Framer Motion's listeners.

### Implementation Steps

#### 1. The Container
Use standard `drag` props for best performance and feel.

```tsx
<motion.div
  drag
  dragMomentum={false} // Optional: prevents sliding after release
  // ... other props
>
  {/* Content */}
</motion.div>
```

#### 2. The Input Wrapper
Wrap your `Input` component in a `div` and attach native listeners via a callback ref.

```tsx
<div
  ref={(node) => {
    if (node) {
      // Define the stop function
      const stop = (e: Event) => e.stopPropagation();
      
      // Attach native listeners to catch ALL touch/mouse starts
      node.addEventListener('pointerdown', stop);
      node.addEventListener('mousedown', stop);
      node.addEventListener('touchstart', stop);
      
      // Cleanup function (called when node changes or unmounts)
      return () => {
        node.removeEventListener('pointerdown', stop);
        node.removeEventListener('mousedown', stop);
        node.removeEventListener('touchstart', stop);
      };
    }
  }}
  // Keep React handlers as a fallback/safety measure
  onPointerDown={(e) => e.stopPropagation()}
  onMouseDown={(e) => e.stopPropagation()}
  onTouchStart={(e) => e.stopPropagation()}
>
  <Input 
    placeholder="You can now select text here!" 
    // ... input props
  />
</div>
```

## Why This Works

- **Native vs. Synthetic**: By using `node.addEventListener`, we interact directly with the DOM event system, bypassing React's synthetic event layer. This ensures we intercept the event *before* it bubbles up to any parent elements (like the `motion.div`) that might have their own native listeners.
- **Comprehensive Blocking**: We block `pointerdown`, `mousedown`, and `touchstart` to cover all possible input methods (mouse, touch, stylus) and browser behaviors.
- **Smoothness Preserved**: Since we don't disable the drag listener on the parent, the dragging physics remain completely native and smooth. We simply "hide" the input interactions from the drag system.
